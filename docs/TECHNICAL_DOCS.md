## 2. Smart Contracts

### 2.1 HostRegistry.sol

**Purpose:** On-chain registry of storage hosts. Maintains economic accountability through security deposits.

**State Variables:**
```solidity
struct Host {
    address addr;
    uint256 spaceGB;
    uint256 pricePerGBPerDay;   // in cents (USD)
    uint256 deposit;             // in wei
    uint256 reputationScore;     // 0–100, init 50
    bool isActive;
    string bankDetails;          // off-chain payout info
    uint256 registeredAt;
}

mapping(address => Host) public hosts;
address[] public hostList;
uint256 public constant MIN_DEPOSIT = 0.01 ether;
```

**Key Functions:**

| Function | Visibility | Modifiers | Description |
|---|---|---|---|
| `registerHost(spaceGB, price, bank)` | external payable | — | Register new host; requires ≥ 0.01 ETH |
| `updateHostInfo(spaceGB, price, bank)` | external | onlyRegisteredHost | Update capacity/pricing |
| `setActiveStatus(active)` | external | onlyRegisteredHost | Pause/resume accepting files |
| `slashDeposit(host, amount, reason)` | external | onlyOwner | Penalise misbehaving host |
| `updateReputation(host, delta, increase)` | external | onlyOwner | Adjust score ±delta |
| `getActiveHosts()` | external view | — | Returns all active host addresses |
| `getHostInfo(addr)` | external view | — | Returns full Host struct |

**Events:** `HostRegistered`, `HostSlashed`, `ReputationUpdated`, `HostStatusChanged`

**Design decision:** Requiring a deposit (even 0.01 ETH ≈ $25) prevents trivial Sybil registrations. The `slashDeposit` function uses `payable(owner).transfer()` so slashed funds go to the platform, not burned — enabling refunds to affected renters in a future iteration.

### 2.2 FileRegistry.sol

**Purpose:** Immutable file ownership ledger. Records the shard hash map and wrapped encryption key.

**State Variables:**
```solidity
struct FileRecord {
    string fileName;
    uint256 totalSize;
    bytes32[] chunkHashes;       // SHA-256 per shard
    address[] hostAssignments;   // which host holds which shard
    address owner;
    uint256 uploadedAt;
    uint8 dataShards;
    uint8 parityShards;
    string encryptedKeyData;     // base64(IV || AES-GCM(fileKey))
    bool exists;
}

mapping(bytes32 => FileRecord) public files;  // key = keccak256(owner, fileName, ts)
bytes32[] public fileIds;
mapping(address => bytes32[]) public ownerFiles;
```

**Key Functions:**

| Function | Visibility | Modifiers | Description |
|---|---|---|---|
| `uploadFileMap(...)` | external | — | Record new file; emits FileUploaded |
| `deleteFile(fileId)` | external | fileExists, onlyFileOwner | Mark file deleted (does not free storage) |
| `verifyOwnership(fileId, addr)` | external view | fileExists | Returns bool |
| `getChunkHashes(fileId)` | external view | fileExists | Returns bytes32[] |
| `getFileRecord(fileId)` | external view | fileExists | Returns full struct |
| `getOwnerFiles(addr)` | external view | — | Returns fileId array |

**Events:** `FileUploaded`, `FileDeleted`

**Design decision:** File IDs are `keccak256(abi.encodePacked(owner, fileName, block.timestamp))`. This allows the same wallet to re-upload a file with the same name (produces a new ID). The `exists` bool in the struct allows logical deletion without an expensive `delete` on a mapping.

### 2.3 HeartbeatMonitor.sol

**Purpose:** Track host liveness via periodic Merkle root submissions.

**State Variables:**
```solidity
struct HeartbeatRecord {
    uint256 lastHeartbeat;
    bytes32 lastMerkleRoot;
    uint256 heartbeatCount;
    bool isBeingMonitored;
}

mapping(address => HeartbeatRecord) public heartbeats;
uint256 public constant HEARTBEAT_TIMEOUT = 1 hours;
```

**Key Functions:**

| Function | Visibility | Modifiers | Description |
|---|---|---|---|
| `startMonitoring(host)` | external | onlyOwner | Initialise monitoring for host |
| `submitHeartbeat(merkleRoot)` | external | onlyRegisteredHost, onlyActiveHost | Record liveness proof |
| `isHostOnline(host)` | external view | — | Returns true if last HB < 1 hour ago |
| `getHeartbeatRecord(host)` | external view | — | Returns full record |

**Design decision:** The Merkle root is self-reported — hosts construct it from their stored shards. This is an optimistic scheme: fraud is detectable when a renter downloads a shard that doesn't match the on-chain hash, triggering a slash. A cryptographic PoRep is a future enhancement.

### 2.4 PaymentLedger.sol

**Purpose:** Pre-paid credit system for rental payments. All values in US cents (integer arithmetic, no floats).

**State Variables:**
```solidity
mapping(address => uint256) public renterBalance;     // cents
mapping(address => uint256) public hostEarnings;      // cents
mapping(address => bool) public registeredHosts;
uint256 public constant PLATFORM_FEE_PERCENT = 5;
uint256 public settlementThreshold = 10000;           // $100
```

**Key Functions:**

| Function | Visibility | Modifiers | Description |
|---|---|---|---|
| `addCredit(renter, amount)` | external | onlyOwner | Top-up renter balance |
| `registerHost(host)` | external | onlyOwner | Whitelist a host for payments |
| `chargeStorageUsage(renter, host, amount, fileId)` | external | onlyOwner | Daily billing |
| `getRenterBalance(addr)` | external view | — | Returns cents |
| `getHostEarnings(addr)` | external view | — | Returns cents |
| `confirmSettlement(host)` | external | onlyOwner | Reset earnings after payout |

**Events:** `CreditAdded`, `StorageCharged`, `PaymentDue`, `SettlementConfirmed`

**Design decision:** `onlyOwner` on `chargeStorageUsage` means the platform controls billing cadence. This is a centralised trust point — a future version would use a Chainlink Automation keeper to call this function permissionlessly on a schedule.

### 3.4 pages/RenterDashboard.jsx

**State:** `balance` (cents from PaymentLedger), `amount` (top-up amount in dollars)

**Razorpay flow:**
1. User enters dollar amount → clicks "Add Credit"
2. POST `/api/create-order` → gets Razorpay order ID
3. Opens Razorpay checkout modal
4. On success: POST `/api/verify-payment` with signature
5. Server verifies HMAC-SHA256 → calls `addCredit` on-chain
6. Frontend polls `getRenterBalance` to show updated balance

**Low balance warning:** Shows at < $0.50 (50 cents).

**Pricing display:** "$0.20/GB/day. 50 GB costs $10/day" — helps users understand burn rate.

---

### 3.5 pages/UploadFile.jsx

**10-step upload flow:**
1. Select file from disk
2. Read as ArrayBuffer
3. Generate fresh AES-256-GCM key
4. Encrypt file → `{ iv, ciphertext }`
5. **Derive wrapping key** from wallet (HKDF via `eth_sign`)
6. **Wrap AES key** → `wrappedKeyB64` (stored in localStorage AND passed to blockchain)
7. Compute SHA-256 of each shard
8. RS(10,4) encode → 14 shards
9. Assign shards to active hosts (round-robin)
10. Call `fileRegistry.uploadFileMap(...)` with all 14 hashes + `encryptedKeyData`

**localStorage key:** `decentrastore_wrappedkey_${fileId}` — fallback for offline decryption if blockchain is unavailable, but the on-chain copy is the source of truth.

---

### 3.6 pages/MyFiles.jsx

**Fetches:** `fileRegistry.getOwnerFiles(wallet.address)` → array of fileIds → parallel `getFileRecord` calls.

**Download flow:**
1. Get `encryptedKeyData` from on-chain `FileRecord`
2. Derive wrapping key from wallet (`deriveWrappingKey`)
3. Unwrap AES key (`unwrapFileKey`)
4. Fetch shards from host URLs (simulated — real P2P layer is future work)
5. RS decode (handles up to 4 missing)
6. AES-256-GCM decrypt → original file bytes
7. Trigger browser download via `URL.createObjectURL(blob)`

## 4. Utility Modules

### 4.1 chunking.js — Reed-Solomon GF(256) Codec

**File:** `frontend/src/utils/chunking.js`
**Size:** ~350 lines, zero dependencies

#### GF(256) arithmetic

```js
const GF_EXP = new Uint8Array(512);  // precomputed powers of generator
const GF_LOG = new Uint8Array(256);  // discrete log table
const PRIM = 0x11D;                  // x^8 + x^4 + x^3 + x^2 + 1
```

`gmul(a, b)` — multiplication via log/exp tables (O(1))
`ginv(a)` — multiplicative inverse: `GF_EXP[255 - GF_LOG[a]]`

#### Cauchy Generator Matrix

```js
function cauchyElement(i, j) {
    // x_i = i, y_j = k + j  (ensures x_i ≠ y_j)
    return ginv(i ^ (RS_DATA_SHARDS + j));
}
```

The full generator matrix is `[I_k | C]` where `C` is the Cauchy matrix. Any `k×k` sub-matrix of this is invertible — the MDS property that guarantees recovery from any `k` shards.

#### Encode

```js
export function encodeShards(data) {
    // 1. Pad data to multiple of RS_DATA_SHARDS
    // 2. Split into k data shards
    // 3. Matrix-multiply G × data_column for each byte position
    // Returns: Uint8Array[RS_TOTAL_SHARDS]
}
```

#### Decode

```js
export function decodeShards(shards) {
    // shards: array of RS_TOTAL_SHARDS items, nulls for missing
    // 1. Find available indices (avail), take first k
    // 2. Extract sub-matrix G_avail
    // 3. Compute G_avail^{-1} via Gauss-Jordan over GF(256)
    // 4. Multiply G_avail^{-1} × received_shards → original data shards
    // Returns: Uint8Array (original data)
}
```

**Why pure JS, not WASM?** The `reed-solomon-erasure` npm package uses WebAssembly with `SharedArrayBuffer` which requires `Cross-Origin-Isolation` headers that conflict with Razorpay's iframe checkout. Rewriting in pure JS eliminated the dependency entirely.

### 4.2 encryption.js — AES-256-GCM + HKDF Key Wrapping

**File:** `frontend/src/utils/encryption.js`

#### Core Functions

```js
generateAESKey()          // returns CryptoKey (AES-256-GCM, extractable)
encryptFile(key, buffer)  // returns { iv: Uint8Array(12), ciphertext: ArrayBuffer }
decryptFile(key, iv, ct)  // returns ArrayBuffer
exportKey(key)            // returns ArrayBuffer (raw 32 bytes)
importKey(rawBytes)       // returns CryptoKey
sha256(buffer)            // returns hex string
```

#### Key Wrapping Functions

```js
async deriveWrappingKey(signer)
```
1. Signs `"DecentraStore key derivation v1 — do not use for transactions"` via `signer.signMessage()`
2. Encodes signature as UTF-8 bytes (the hex string)
3. Imports as HKDF key material
4. `deriveBits(HKDF-SHA256, salt="decentrastore-wrap-salt-v1", info="aes-wrapping-key", 256)`
5. Imports result as AES-GCM `wrapKey`
6. Returns `CryptoKey`

```js
async wrapFileKey(fileKey, wrappingKey)
```
1. Generate random 12-byte IV
2. `SubtleCrypto.wrapKey("raw", fileKey, wrappingKey, {name:"AES-GCM", iv})`
3. Prepend IV to wrapped bytes
4. Return base64 string (56 chars)

```js
async unwrapFileKey(wrappedB64, wrappingKey)
```
1. Decode base64 → bytes
2. Extract IV (first 12 bytes), wrapped key (remaining)
3. `SubtleCrypto.unwrapKey("raw", wrappedBytes, wrappingKey, {name:"AES-GCM", iv}, {name:"AES-GCM"}, true, ["encrypt","decrypt"])`
4. Returns `CryptoKey`

## 5. Payment Server

**File:** `server/index.js`

### Endpoints

#### `POST /api/create-order`
```json
Request:  { "amountDollars": 10, "walletAddress": "0x..." }
Response: { "orderId": "order_xxx", "amount": 1000, "currency": "USD", "keyId": "rzp_..." }
```
- Converts dollars → cents (integer)
- Creates Razorpay order via `razorpay.orders.create()`

#### `POST /api/verify-payment`
```json
Request:  { "orderId": "order_xxx", "paymentId": "pay_xxx", "signature": "hmac_hex",
            "walletAddress": "0x...", "amountCents": 1000 }
Response: { "success": true, "txHash": "0x..." }
```
- Verifies: `HMAC-SHA256(razorpay_order_id + "|" + razorpay_payment_id, secret) === signature`
- If valid: connects to contract via provider, calls `paymentLedger.addCredit(wallet, cents)`

#### `GET /api/balance/:address`
```json
Response: { "balance": 1000 }
```
- Returns `paymentLedger.getRenterBalance(address)` as number

#### `GET /api/health`
```json
Response: { "status": "ok" }
```
Used by Render.com health check.

### CORS Policy
Accepts: `http://localhost:5173`, `http://localhost:4173`, `*.vercel.app`, `process.env.FRONTEND_URL`

## 8. Design Decisions

### Why 4 contracts instead of 1?

Separation of concerns + independent upgradeability. HostRegistry can be upgraded without touching payment logic. Each contract has a focused, auditable surface area. Gas cost per deploy is higher, but operational clarity and maintainability are worth it.

### Why USD cents instead of ETH/wei for payments?

Storage pricing in ETH is volatile — a host who set $0.20/GB/day when ETH=$2000 would earn very different real money if ETH drops to $500. USD-denominated pricing (stored as cents integers) provides renters and hosts with predictable economics. The on-chain contract stores only integer cent amounts; conversion to ETH/fiat happens at settlement time.

### Why Razorpay instead of on-chain ETH payment?

Most renters don't have ETH. Razorpay (supporting credit/debit cards, UPI, net banking) dramatically lowers the barrier to entry. The security model is: server verifies Razorpay's HMAC-SHA256 webhook signature before calling `addCredit` — no fake credits possible without Razorpay's secret key.

### Why pure-JS Reed-Solomon instead of WASM?

The `reed-solomon-erasure` package uses `SharedArrayBuffer`, which requires `Cross-Origin-Isolation: require-corp` and `Cross-Origin-Opener-Policy: same-origin` HTTP headers. These headers break Razorpay's payment iframe (a cross-origin resource). The pure-JS implementation runs without any special headers, has no native dependencies, and for file sizes up to a few MB is fast enough (11ms for 1MB).

### Why HKDF+AES-GCM key wrapping instead of password-based encryption?

Users already have an Ethereum wallet — an existing PKI. Deriving a wrapping key from `eth_sign` output:
- Requires no additional password to remember
- Is deterministic (same wallet → same wrapping key → can always unwrap)
- Is wallet-locked (only the holder of the private key can sign the derivation message)
- Works offline (no server involved in key derivation)

The alternative (user password) would require storing a password hash and dealing with password resets — unnecessary complexity given wallet availability.

## 9. Security Analysis

### Threat: Malicious Host

A host could serve corrupted shards. **Mitigation:** Each shard's SHA-256 is stored on-chain. The downloader verifies every shard before passing to RS decoder. A mismatch triggers a UI warning and can be reported to the admin for `slashDeposit`.

### Threat: Blockchain Observer

All data on Ethereum is public. `encryptedKeyData` is public. **Mitigation:** The wrapped key is useless without the wallet's `eth_sign` output. An observer cannot decrypt the file key without the wallet private key.

### Threat: Compromised Payment Server

If the server is hacked, attacker could call `addCredit` for any address. **Mitigation:** `addCredit` is `onlyOwner` — the server's private key is the deployer key. Monitoring the deployer key for unexpected `addCredit` calls provides detection. In production, the deployer key should be a hardware wallet multisig.

### Threat: Fake Razorpay Webhook

An attacker posts a fake `/api/verify-payment` with invented data. **Mitigation:** The server computes `HMAC-SHA256(orderId + "|" + paymentId, razorpaySecret)` and compares to the submitted signature. Forgery requires knowledge of the Razorpay secret key, which is server-side only.

### Threat: Replay Attack on Key Derivation

Could a malicious dApp trick a user into signing the same derivation message and obtaining their wrapping key? **Mitigation:** The wrapping key is only useful for decrypting the on-chain `encryptedKeyData`. The attacker would still need the on-chain blob. This is equivalent to saying: if the attacker has both the signature and the on-chain key blob, they can decrypt the file key. Since both are required simultaneously, the user's risk surface is limited to malicious dApps that can silently call `eth_sign` (mitigated by MetaMask's confirmation dialog).

## 10. Known Limitations

| Limitation | Impact | Planned Fix |
|---|---|---|
| No actual P2P shard transfer | Files can't be retrieved from hosts; currently simulated | libp2p integration |
| Self-reported Merkle roots | Host can claim to store shards it doesn't have | PoRep challenge-response |
| Admin-controlled billing | `chargeStorageUsage` relies on trusted caller | Chainlink Automation |
| No shard re-replication | If a host goes offline permanently, shard is lost | Background re-replication daemon |
| Single deployer key | Payment server uses same key as contract owner | Multisig with Gnosis Safe |
| Local shard storage | No actual distributed storage layer | S3/IPFS per-host storage backend |
